# Harness多租户恢复与Binding V2边界完成

时间：2026-07-15 00:02（Asia/Shanghai）

## 粗粒度事件

Runtime进入P0.5/P0.6公共执行基座收口后，Harness同步完成了面向6+1和未来自定义组件的预防性加固。该事件只记录Harness公共边界变化，不代表Runtime/Application治理桥已经完成。

## 已落地边界

- Endpoint与Run Session按完整Execution Scope+Run分区，不再使用裸Instance或Run ID；不同租户可复用tenant-local Identity、Lineage、Instance、Sandbox和Run ID；
- Event source identity绑定完整Run作用域，同一Instance顺序运行多个Run时各自从source sequence 1开始而不冲突；
- Event Candidate append出现任何错误（包括原生timeout/cancel）后都只按source identity/epoch/sequence Inspect，同序换内容继续冲突；
- 模型调用返回错误或调用后Evidence持久化失败时进入`waiting_reconciliation`，不产生虚假终态，也不盲重派Effect；
- Provider Evidence Digest和native session进入明确Observation；Runtime stable session、Harness稳定session和Provider native session三者永久分离；
- Runtime Adapter可显式接收`AdapterBindingManifest`并实现`DescriberV2`，但不会从legacy Manifest推导Locality、Owner、Residual或dispatch资格；
- 已注入V2治理身份的宿主Adapter主动拒绝legacy Open/Control/Close，等待P0.6双Binding与Delegation治理桥；
- Endpoint关闭后只允许`cleanup`恢复观察，不能继续Ready/State/Event/Control；该观察不是Runtime权威Cleanup事实；
- Harness私有ContextPort只允许消费已物化Snapshot，不能成为未记账的远程检索、披露或Cache写入后门；
- Opaque Payload、Manifest、Run、Event、Context、Action和native session均补充确定性容量上限。

## 已验证反例

- 跨Endpoint、跨Scope、跨租户控制和读取拒绝；
- 同租户单Instance单活跃Run与跨租户并发Run隔离；
- 同Instance顺序Run不复用Evidence source；
- 自定义第八Harness Adapter及同kind多实现只可成为Certification Candidate，不能自授Binding、Production或Dispatch；
- Event回包丢失、模型回包未知、调用后Evidence故障、Cancel迟到结果、关闭后恢复观察均保持fail closed；
- Contract和Control Payload已执行独立fuzz，不依赖具体6+1组件枚举。

## 当前停止线

宿主Execution Adapter与Data Plane Harness Provider必须是两个独立Binding身份；P0.6需提供显式Delegation、Operation Scope、Governance Permit、实际执行点复读与Enforcement Receipt。该公共桥完成前，Harness V2身份只能发现和认证候选，不能执行生产动作。
