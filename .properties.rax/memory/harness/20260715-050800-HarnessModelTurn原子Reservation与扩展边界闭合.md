# Harness Model Turn原子Reservation与扩展边界闭合

Harness已在Application进入Runtime治理前增加持久Domain Reservation。首次Reservation在注入Clock下复读精确Candidate、Intent、Session revision与Adapter；Harness同一Fact Owner用一个原子提交同时推进Session `waiting_model_dispatch→model_dispatch_reserved`并建立Attempt恢复索引。同一Session+Candidate在64个并发Attempt下只有一个赢家，回包丢失后只允许Inspect，不产生ghost索引或双主。

Reservation expiry取Candidate与Intent最小值。首次创建必须current；已提交Fact即使随后过期仍可按Attempt历史读取，使未知回包和Settlement能够收口，但Application/Runtime在dispatch前继续fail closed。pre-prepared unknown只能从精确Reservation失败终止，BindPrepared必须携带并复读同一Reservation。

Application专用Reservation/Binding类型已移入`harness/bridgecontract`；Harness核心`contract`只保存传输无关的本地Reservation投影，并有自动导入边界测试禁止反向依赖Application。合法namespaced自定义Step可通过同一版本化Port与自己的exact Adapter配置接入，无需给Runtime/Harness增加kind switch。

本切面已覆盖普通全量、Reservation 100轮、Race 20轮、全量Race、Vet、lost reply、首次过期、terminal/ref-only零变更、深拷贝和真实Application Coordinator黑盒闭环。Fake只证明合同/CAS/并发语义，不声明生产持久性或SLA。
