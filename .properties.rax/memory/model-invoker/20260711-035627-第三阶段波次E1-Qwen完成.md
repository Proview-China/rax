## 官方合同结论

- 当前只批准中国北京 `cn-beijing`与新加坡 `ap-southeast-1`的 Workspace专属按量 Endpoint；API Key、Workspace、Region和模型列表不能跨区复用；
- Responses为默认主路径，Chat Completions为补充路径；DashScope原生、Anthropic兼容、内置工具、多模态、Batch、异步与其他 Region继续延期；
- 按量新 Key使用 `sk-ws-*`，旧 `sk-*`仍有效；Coding/Token Plan使用更具体的 `sk-sp-*`和专属 Endpoint，不能进入按量 Adapter；
- 当前北京/新加坡共同文本白名单为八个稳定 Qwen别名；日期快照、模态和未知模型不进入 callable集合；
- Responses只处理文档列出的字段，未列字段会被服务端静默忽略，因此 Adapter必须预调用拒绝而不能把忽略冒充成功映射。

## 实际产物

1. 新增 `provider/qwen/`独立 Runtime Adapter，默认 Responses并补充 Chat；生产 Endpoint只由 Region与 Workspace确定性生成，BaseURL覆盖只允许 loopback测试；
2. Responses保留 typed message/function/reasoning output、usage、7天 `previous_response_id` State与 SSE；Chat把 portable reasoning映射为 `enable_thinking`/`thinking_budget`并保留非流/流 `reasoning_content`与 JSON Object；
3. 未批准的 metadata、parallel控制、strict、Responses structured output、Chat developer role和 thinking工具历史不完整均在 HTTP前拒绝；
4. Credential Profile新增可选 `denied_key_prefixes` Schema字段；拒绝前缀优先于允许前缀，准确表达 `sk-*`按量兼容但 `sk-sp-*`订阅禁止；
5. 新增北京/新加坡 × Responses/Chat四条 `implemented_offline + callable=true` Binding；Catalog由57条扩展为61条：38条 callable、23条控制记录，对应十三个 Runtime Adapter；
6. 原 Alibaba Coding Plan与 Token Plan六条控制记录继续 `interactive_coding_only + planned + callable=false`，没有 Adapter且不会自动切换余额或 Endpoint；
7. 新增 `tests/qwen/`黑盒与 fuzz、`tests/integration/qwen_smoke_test.go`显式入口；统一脚本清空 Qwen与此前遗漏的 MiMo烟测开关后只编译 integration。
