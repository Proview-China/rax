## 新确认或强化的设计

1. 每种Harness必须在外部语义上满足Bootstrap、Run Controller、Interaction Loop、Session、Model/Tool协调、Context接入、事件、控制、Checkpoint、错误、背压、健康和清理等共用合同；内部可以复用Praxis内核，也可以由官方/第三方Adapter等价实现；
2. 使用者Profile、开发者ComponentProfile和执行Resolved Profile分层，Runtime只读取不可变解析结果；
3. 扩展体系采用稳定公共语义、Capability Descriptor、Provider/Adapter合同、命名空间扩展、Extension Registry和合同测试，不把官方Sandbox或单机实现写死；
4. User SDK与Extension SDK分离，REPL、Remote API和User SDK共享Application Command API；
5. Model Profile继续按精确Provider、Model revision、Deployment、Protocol、Offering/Auth Route、Execution Surface和Harness栈选择，并合成ModelBehaviorProfile、HarnessCapabilityProfile和RuntimePolicy；
6. Context Engine内置Cache Manager，负责稳定前缀、CachePlan、键、TTL、失效、Affinity和节省证据；Model Profile声明Route缓存能力，Invoker做原生映射，Runtime只维持Affinity与因果关联；
7. 当前单机与未来Praxis Agent Cluster共享Lease、PlacementRef、ResourceClaim和Capability语义。
