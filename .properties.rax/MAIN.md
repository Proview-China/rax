# `.properties.rax` 构建指南

## 1. 文件定位

`.properties.rax` 是本仓库的设计与构建控制中心。所有主题先在这里完成设计、确认、计划和验收定义，再进入代码实现。

本文件负责说明统一的构建方法。执行任何模块前，应先阅读本文件和仓库根目录的 `AGENTS.md`；两者冲突时，以 `AGENTS.md` 为准。

## 2. 核心原则

1. 先设计，后计划，再实现；未通过确认，不进入下一阶段。
2. 用户负责项目方向、模块设计和关键决策；Codex 负责整理、翻译、执行、测试和结果同步。
3. 不允许未经确认自行新增模块、扩大范围或改变技术边界。
4. 所有重要资产必须可以自动验证；不能只以“文件已生成”作为完成标准。
5. 所有需要用户阅读和确认的内容使用中文。
6. 目录名和模块名必须在 `design`、`plan`、`memory`、`module` 中保持一致。
7. `.properties.rax` 保存项目级控制资产；真实代码只在计划批准后写入计划指定的位置。

## 3. 目录职责

| 路径 | 职责 | 主要内容 |
|---|---|---|
| `MAIN.md` | 构建总指南 | 阶段、规则、验收和同步方法 |
| `design/<模块名>/` | 设计事实源 | draw.io、Excalidraw、中文设计说明、边界和关键决策 |
| `plan/<模块名>/` | 落地计划 | 产物清单、粗粒度阶段、细粒度任务、风险和验收标准 |
| `memory/<模块名>/` | 项目进度快照 | 按时间保存的粗粒度事件、当前状态、决定和阻塞项 |
| `module/<模块名>/` | 模块说明入口 | 模块作用、组成、边界、使用方式、测试和已知限制 |
| `properties/` | 项目总体索引 | 总体进度、架构概述、模块目录和重要产物索引 |

`properties/` 只描述项目整体，不放置某个模块的详细设计、计划或实现说明。模块细节分别进入同名的 `design`、`plan`、`memory` 和 `module` 目录。

## 4. 标准构建流程

```text
主题提出
  -> 设计（draw.io / Excalidraw）
  -> 中文设计说明
  -> 用户确认设计
  -> 详细计划
  -> 用户审核计划与预期产物
  -> 代码实现
  -> 自动化测试与验收
  -> 模块说明和项目状态同步
```

### 阶段一：设计

在 `design/<模块名>/` 中完成：

- 原始 draw.io 或 Excalidraw 文件；
- 设计目的和要解决的问题；
- 模块边界、输入、输出和依赖；
- 关键流程、异常路径和性能要求；
- 尚未确定的问题和需要用户决定的事项。

图形资产必须翻译为可执行的中文说明，不能只保存图片。设计未获得用户确认前，不创建真实模块代码。

### 阶段二：计划

设计确认后，在 `plan/<模块名>/` 中产生唯一对应的落地计划。计划至少包含：

- 明确范围和不做事项；
- 最终会产生的文件、接口、命令或服务；
- 粗粒度阶段清单；
- 可逐项执行的细粒度任务清单；
- 技术选型、依赖、风险和回退方法；
- 单元测试、集成测试、黑盒测试和白盒测试要求；
- 完成条件和用户可预见的最终结果。

计划完成后必须先交给用户审核。实现完成后不删除计划，将其保留并标记为“陈旧计划”，作为历史依据。

### 阶段三：实现

只有设计和计划都已确认，才可以开始实现：

- 严格按照批准范围落地；
- 新增模块或改变边界前重新确认；
- 不顺手重构无关内容；
- 代码、配置、接口和自动化流程必须与计划一致；
- 高性能和高度优化必须建立在可测量、可验证的指标上。

### 阶段四：测试与验收

每个模块必须根据风险完成相应验证：

- 重要逻辑具备充分的单元测试；
- 模块内部执行白盒测试；
- 面向外部行为执行黑盒测试；
- 模块组合和关键链路执行集成测试；
- 对最终产物执行真实运行或可用性验证；
- 记录测试命令、结果、失败原因和未覆盖范围。

集成测试由用户决定执行时机。测试未通过或证据不完整时，不得标记为完成。

### 阶段五：说明与同步

模块落地后必须完成：

1. 在 `module/<模块名>/` 中提供不阅读源码也能理解的中文说明；
2. 在 `memory/<模块名>/` 中记录本次粗粒度进展；
3. 在 `properties/` 中更新项目总体状态和模块索引；
4. 确认设计、计划、实现、测试和说明之间不存在明显偏差；
5. 再执行 Git 提交与远端同步。

## 5. 模块命名与文件约定

同一模块必须使用一个稳定的 `<模块名>`：

```text
.properties.rax/
|-- design/<模块名>/
|-- plan/<模块名>/
|-- memory/<模块名>/
`-- module/<模块名>/
```

- 模块目录名使用简短、稳定的英文小写名称；需要多个单词时使用短横线。
- 用户可见的标题、说明和验收内容使用中文。
- `memory` 文件使用本地时间戳：`YYYYMMDD-HHmmss-事件名称.md`。
- 时间快照记录阶段变化、关键决定、验收结果或重大阻塞，不记录每次微小文件修改。
- 图形、文档和测试证据应使用可以追溯来源与用途的文件名。

## 6. 模块说明的最低要求

每份 `module/<模块名>/` 说明资产至少回答：

1. 这个模块解决什么问题；
2. 它包含哪些组成部分；
3. 它接收什么输入并产生什么输出；
4. 它依赖什么，又被什么依赖；
5. 如何构建、运行、配置和验证；
6. 已通过哪些测试；
7. 当前限制、风险和后续方向是什么。

模块说明是模块的第一入口，必须能够帮助读者在不先阅读源码的情况下理解和使用该模块。

## 7. 技术职责边界

| 技术 | 默认职责 |
|---|---|
| Go | 项目核心、主要服务和稳定的内核能力 |
| TypeScript | 整合层、工具层、接口适配和 glue 代码 |
| Rust | 计算密集、性能敏感且适合独立优化的部分 |

具体模块是否使用某种技术，仍需由设计和计划确认，不能仅因为默认职责而自动新增实现。

## 8. 开始执行前检查

开始任何真实构建前，必须确认：

- [ ] 模块名称唯一且各目录一致；
- [ ] draw.io 或 Excalidraw 设计已保存；
- [ ] 图形内容已翻译为中文设计说明；
- [ ] 用户已确认设计和模块边界；
- [ ] 详细计划已完成并通过审核；
- [ ] 预期产物和验收结果已向用户说明；
- [ ] 测试范围、命令和完成标准明确；
- [ ] 实现位置和技术职责已经确定。

任何一项缺失，都应停留在当前阶段补全，不得直接进入代码实现。
